# Hermes 观测台 — 设计文档 v3

> **定位**：横切在所有现有进化机制之上的**观测与编排层**，不替代任何现有机制，只做三件事：采集进化事件 → 织成进化谱系 → 呈现进化叙事。
>
> **v3 新增**：变化探测与自适应适配机制——当 Hermes 进化机制本身发生变化时，观测台能自动检测、告警、适配，而不需要手动改代码。

---

## 1. 核心设计决策

### 只读旁路

观测台是"只读旁路"的——它只采集、分析、呈现，**不直接修改任何进化机制**。这样既不会引入新的故障面，也保证观测本身不影响被观测对象。

| 做什么 | 不做什么 |
|--------|---------|
| 只读 `~/.hermes/` 下所有文件 | 不写入任何 Hermes 文件 |
| 通过 Plugin Hook 观察事件 | 不拦截或阻止任何工具调用 |
| 审批操作转发给 Hermes 原生 API | 不自行执行审批逻辑 |
| 独立进程运行 | 崩溃不影响 Agent |

## 2. 四判据检验

| 判据 | 实现 |
|------|------|
| **可观测** | Plugin Hook 采集所有进化事件 → `evolution-events.jsonl` → WebSocket 实时推送 → 时间线视图 |

## 3. 变化探测与自适应适配（v3 新增）

### 3.1 问题

Hermes 的进化机制不是一成不变的。它会随版本更新而演进——新增配置项、变更文件结构、修改事件格式、引入新的工具 action。如果观测台硬编码了所有数据格式，一旦 Hermes 更新，观测台就会失效。

### 3.2 五层探测策略

从版本号到语义，逐层深入：

| # | 探测层 | 检测内容 | 数据来源 | 频率 |
|---|--------|---------|---------|------|
| 1 | **版本探测** | Hermes 版本号变化 | `GET /api/status`（返回 agent version） | 每次启动 |
| 2 | **配置 Schema 探测** | 配置项新增/删除/类型变更 | `GET /api/config/schema`（返回每个字段的类型、描述、分类） | 每次启动 |
| 3 | **文件结构探测** | `~/.hermes/` 下新增/消失的文件和目录 | 文件系统扫描 | 每小时 |
| 4 | **事件格式探测** | 工具名变更、新增 action、事件字段变化 | Plugin Hook 事件流（`post_tool` 观察到的工具调用） | 实时 |
| 5 | **语义探测** | 已知数据源的内容格式变化（如 `.usage.json` 新增字段） | JSON Schema 校验 | 每小时 |

### 3.3 六大适配策略

检测到变化后，观测台如何自动适配：

| 策略 | 说明 | 示例 |
|------|------|------|
| **插件化采集器** | 每个数据源是独立插件。新增/修改一个不影响其他。通过 `collectors/` 目录注册。 | MEMORY.md 采集器、.usage.json 采集器、GEPA metrics 采集器各自独立 |
| **Schema 注册表** | 维护所有已知数据源的 Schema。检测到未知 Schema 时告警，不崩溃。 | `.usage.json` 新增 `last_evolved_at` 字段 → 告警但继续读取 |
| **自适应渲染** | 前端根据事件 Schema 动态渲染字段，不硬编码。未知字段显示为 `[unknown field]`。 | `skill_manage` 新增 `archive` action → 自动出现在时间线中 |
| **版本兼容层** | 后端有版本适配层，将不同版本的格式统一为内部标准格式。 | Hermes v2.3 的 `.usage.json` 没有 `pinned` 字段 → 适配层补默认值 `false` |
| **降级模式** | 某个数据源不可读时，对应视图显示"数据源不可用"而非整个系统崩溃。 | `curator/logs/` 目录不存在 → Curator 视图显示"无 Curator 运行记录" |
| **变更日志** | 记录每次检测到的变化，形成观测台自身的"进化日志"。 | "v2.4.0 → v2.4.1: 新增 curator.consolidate_threshold 配置项" |

### 3.4 变化类型与影响分级

| 变化类型 | 严重度 | 观测台影响 | 适配方式 |
|---------|--------|-----------|---------|
| 新增配置项 | 低 | 无需修改 | 自适应渲染自动显示 |
| 新增文件/目录 | 低 | 无需修改 | 文件结构探测记录，新采集器可选注册 |
| 新增工具 action | 低 | 无需修改 | 自适应渲染自动识别 |
| 配置项类型变更 | 中 | 需检查 | Schema 注册表告警，人工确认 |
| 文件路径变更 | 高 | 需适配 | 采集器插件更新路径 |
| 事件格式不兼容 | 高 | 需适配 | 版本兼容层编写适配规则 |
| 机制移除 | 高 | 需适配 | 对应视图进入降级模式 |

## 4. 多 Profile 支持

Hermes 支持多 Profile（`hermes -p coder`、`hermes -p research`），每个 Profile 有独立的 `HERMES_HOME`。观测台需要：

- **Profile 选择器**：侧边栏顶部显示所有可用 Profile，点击切换
- **每条事件标注 Profile**：时间线、技能库、记忆状态中的每条数据都标注所属 Profile
- **后端 `?profile=` 路由**：所有 API 请求携带 Profile 参数，后端路由到对应的 `HERMES_HOME`
- **跨 Profile 对比**：总览仪表盘支持多 Profile 对比视图

## 6. HTML 原型的 12 个视图

1. **总览仪表盘** — KPI 卡片 + 增长趋势图 + 三层进化活动概览
2. **进化时间线** — 所有事件按时间倒序，标注 Profile/触发源/session/turn
3. **记忆状态** — MEMORY.md + USER.md + 容量进度条
4. **技能库** — 技能列表 + 生命周期状态 + 使用统计 + 来源标记
5. **Curator 活动** — 运行历史 + 状态分布 + 安全保证
6. **GEPA 进化管道** — 分数对比 + 帕累托前沿 + 五重护栏
7. **进化谱系** — 版本树 + diff 可视化
8. **待审批** — pending 列表 + approve/reject
9. **进化叙事** — 自动生成故事线 + 量化效果
10. **系统架构** — 三层架构图 + Hook 集成 + 多 Profile 路由
11. **四判据 + 术语表** — 判据对照 + 26 个术语详细解释
12. **变化探测**（v3 新增）— 版本信息 + Drift 检测结果 + 五层探测策略 + 适配策略 + 健康检查
